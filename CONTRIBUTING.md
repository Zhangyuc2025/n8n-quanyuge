# 内部开发协作指南

本仓库为公司内部项目，不再接受外部贡献。所有修改需遵循以下原则：

1. **安全与合规优先**  
   - 禁止向公众渠道泄露代码、文档或构建产物  
   - 严格审查第三方依赖及许可证，新增依赖需通过平台工程组审批

2. **分支与提交规范**  
   - 统一使用公司内部 Git 平台，禁止向外部 fork 推送  
   - 分支命名：`feature/<topic>`、`fix/<issue>`、`chore/<scope>`  
   - 提交信息遵循 Conventional Commits，必要时附带工单编号

3. **代码评审流程**  
   - 至少一名核心维护者 Review 通过后方可合并  
   - 对涉及安全、计费、权限的改动需额外拉平台负责人参与评审  
   - 重要决策同步记录于 `二开改造文档/`

4. **质量保障**  
   - 提交前执行 `pnpm lint`、`pnpm format`、`pnpm test`（若测试尚未修复，请在 MR 中说明）  
   - 新功能需附带相应测试或验收步骤  
   - 影响配置或部署的变更，务必更新相关文档

5. **发布与部署**  
   - 发布流程由平台工程组统一负责，禁止个人私自打包对外  
   - Docker 镜像、二进制包等产物仅可推送至内网仓库  
   - 如需向其他业务团队交付，请随附版本说明与回滚方案

如需了解更多流程或申请权限，请联系平台工程组项目负责人。
